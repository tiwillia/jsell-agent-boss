# MCP Bootstrap Server ‚Äî Replacing `./commands/*` Symlinks

**TASK-059 | Area: (4) MCP server replacing symlink approach**

## Current State: The Symlink Problem

The current bootstrap mechanism relies on symlinks in `./commands/`:

```
commands/
  boss.check.md    -> /home/jsell/.claude/commands/boss.check.md
  boss.ignite.md   -> /home/jsell/.claude/commands/boss.ignite.md
  boss.plan.md     -> /home/jsell/.claude/commands/boss.plan.md
```

These symlinks allow Claude Code to discover slash commands like `/boss.check` and
`/boss.ignite`. Problems:

1. **Fragile**: Symlinks break when the repo is cloned to a different path
2. **tmux-only**: The ambient backend sends initial prompts directly ‚Äî it cannot "type"
   a slash command. There is no equivalent bootstrap mechanism for non-tmux backends
3. **Invisible**: The coordinator has no visibility into what commands are available or
   what version they are
4. **Out-of-band**: Commands live outside the coordinator data model. When the server
   is updated, command files must be manually synced
5. **Not portable**: A user on a fresh machine must manually create the symlinks

## Proposed: Boss MCP Server

Replace the symlink approach with a local MCP (Model Context Protocol) server embedded
in the `boss` binary. The MCP server exposes a single **resource** ‚Äî the bootstrap prompt ‚Äî
that any MCP-compatible AI backend can pull on startup.

### MCP Resource: `boss://bootstrap/{space}/{agent}`

When an AI client fetches this resource, it receives the fully assembled bootstrap prompt:

```
[persona prompts, in order]

[agent initial_prompt]

--- Agent Boss Context ---
Space: AgentBossDevTeam
Agent: LifecycleMgr
Coordinator: http://localhost:8899
Session: agentbossdevteam-lifecyclemgr
```

This replaces the need for slash command files. The AI reads the resource on startup and
knows exactly what to do.

### MCP Server Design

The boss binary adds an MCP endpoint alongside the existing HTTP API:

```
GET /mcp/sse         ‚Äî SSE stream for MCP protocol handshake
POST /mcp/message    ‚Äî JSON-RPC message handler
```

MCP protocol version: 2024-11-05 (current stable).

Capabilities exposed:
- `resources` ‚Äî list and read resources
- No `tools`, no `prompts` (keep it minimal)

Resources listed:

```json
{
  "resources": [
    {
      "uri": "boss://bootstrap/AgentBossDevTeam/LifecycleMgr",
      "name": "Bootstrap prompt for LifecycleMgr",
      "mimeType": "text/plain"
    }
  ]
}
```

### Backend Integration

#### tmux backend

After creating the session and waiting for Claude to initialize, instead of typing
`/boss.ignite "Name" "Space"`, the tmux backend types:

```
claude --dangerously-skip-permissions --mcp-config /tmp/boss-mcp.json
```

Where `/tmp/boss-mcp.json` is generated by the server at spawn time:

```json
{
  "mcpServers": {
    "boss": {
      "url": "http://localhost:8899/mcp/sse",
      "type": "sse"
    }
  }
}
```

Claude starts with the boss MCP server connected. On startup, Claude reads
`boss://bootstrap/{space}/{agent}` and follows the instructions in it (which
include the equivalent of what `/boss.ignite` used to do: read the blackboard,
post initial status, begin work).

Alternatively (simpler first step): tmux backend types the assembled bootstrap
prompt directly, as it does today ‚Äî but the prompt is now fetched from
`GET /spaces/{space}/ignition/{agent}` (which already exists) rather than from
a symlinked file.

#### ambient backend

The ambient backend passes the assembled bootstrap prompt as the `Command` field in
`SessionCreateOpts`. It already does this indirectly. With this change, the coordinator
populates `Command` from `AgentConfig.InitialPrompt` + persona injections, making it
explicit and reproducible.

For MCP-native ambient backends (future): pass the MCP server URL in `AmbientCreateOpts`
and let the backend handle resource fetching natively.

### Slash Command Files ‚Äî Transition Plan

The existing `commands/*.md` files are kept for backward compatibility but marked as
deprecated in a `commands/README.md`. New installations use the MCP approach.

A future cleanup task removes the files after one release cycle.

## MCP Server Implementation Notes

- Pure Go, stdlib HTTP only (no external MCP library needed for this minimal surface)
- SSE transport (not stdio ‚Äî we are server-mode, not subprocess-mode)
- Authentication: none (localhost only; same trust model as existing HTTP API)
- The MCP endpoint is gated behind the same `COORDINATOR_PORT` as the rest of the API

### JSON-RPC Methods Handled

| Method | Description |
| ------ | ----------- |
| `initialize` | Negotiate capabilities |
| `resources/list` | Return available bootstrap resources |
| `resources/read` | Return assembled bootstrap prompt for a given agent |
| `ping` | Health check |

All other methods return `-32601 Method not found`.

## Open Questions

  to minimize configuration.

- **[?BOSS] MCP config generation**: Should `/tmp/boss-mcp.json` be written by the
